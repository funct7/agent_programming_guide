# Programming Guide Index

Use this guide as the global source of truth for programming guidance.

## How To Use

- Start with generated [`catalog.yaml`](./catalog.yaml) before substantive code review or edits.
- Do a minimal target scan first: language, project area, file type, visibility, and obvious constructs.
- Select applicable catalog entries from that scan, then read the source files listed by those entries.
- For code review, use the generated `rules_by_review_pass` index and run review passes in order: structural, substantive, repeated-check.
- After the global guide pass, read any project-local programming guide declared by the current project instructions.
- If code inspection reveals additional constructs, return to the catalog and select additional entries.
- Do not treat this index as a replacement for the actual rules.

## Project-Local Layer

The programming guide is a two-tier system:

1. Global guide: this directory is the source of truth for general programming rules, principles, preferences, and repeated mistakes.
2. Project-local guide: a project may declare local additions, exceptions, or conventions in its own instructions.

Read project-local guidance after the global guide. Treat it as local additions or explicitly scoped exceptions, not as a duplicate source of general rules. If project-local guidance appears to conflict with a global core rule, stop and make the conflict explicit before proceeding.

## Tiers

- [Core Rules](./core-rules.md): small set of rules that should almost never be violated.
- [Principles](./principles.md): default reasoning guidance; follow unless local context gives a good reason not to.
- [Preferences](./preferences.md): Josh's preferred code organization and style; important but more flexible.
- [Conventions](./languages/swift.md): language-specific, framework-specific, or cross-cutting conventions.
- [Repeated Mistakes](./repeated-mistakes.md): recurring agent failure modes to check explicitly before finishing work.

## Authoring Notes

- Metadata lives in YAML code blocks under rule/example headings and case files. Metadata does not use `kind`; the ID prefix and file location imply the record type.
- Example IDs start with `example.`; feedback case IDs start with `case.`.
- `catalog.yaml`, `examples/catalog.yaml`, and `cases/catalog.yaml` are generated by the `programming-guide-maintenance` skill; do not edit them by hand.
- `meta.yaml` defines allowed tiers, review passes, tags, preference strength bounds, and case statuses.
- Keep metadata short and optimized for routing.
- Do not invent new tags or tiers; ask Josh to choose them before updating `meta.yaml`.
- Do not add project-specific layer names to global guide metadata; use constructs, language, tags, and review passes.
- Put nuance, examples, and caveats in the linked Markdown source files.
- Include sample code only when it materially clarifies a rule or principle.
- Do not add sample code when the instruction is already clear enough without it.
